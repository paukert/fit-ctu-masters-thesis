\section{Výběr metrik}
S ohledem na různé druhy činností prováděných během vývoje softwaru si tato práce nedává za cíl vybrat nebo vytvořit metriku, která by se snažila pomocí jediného čísla popsat celkovou výkonnost vývojářů. Dle SPACE frameworku není ani možné produktivitu vývojářů vysvětlovat a měřit pomocí jediné dimenze (metriky). \cite{space} Z tohoto důvodu bylo vybráno několik kvantitativních metrik z oblasti code review a úpravy zdrojového kódu, které pokrývají celkem tři dimenze definované SPACE frameworkem. Jak již bylo zmíněno v sekci \ref{section:space-framework}, tento počet je dle jeho autorů dostatečný pro získání vypovídajících výsledků. Kvantitativní metriky byly zvoleny záměrně, neboť jejich sběr, zpracování a vyhodnocení bývá výrazně jednodušší než u metrik kvalitativních, které často mohou vyžadovat subjektivní interpretaci.

U každého uživatele spojovaného se sledovaným projektem budou evidovány následující metriky, přičemž u každé z nich je v závorce uvedena příslušná dimenze SPACE frameworku, do které spadá:

\begin{enumerate}
    \item \textbf{Počet vytvořených pull requestů} (aktivita)\\
    Počet pull requestů, které daný uživatel vytvořil. Do počtu jsou zahrnuty jak uzavřené, tak mergnuté, ale i aktuálně stále otevřené pull requesty.
    \item \textbf{Počet mergnutých pull requestů} (aktivita)\\
    Počet pull requestů, které daný uživatel vytvořil a které byly zároveň mergnuty do cílové větve. Výrazně odlišný počet od počtu vytvořených pull requestů může indikovat například špatnou kvalitu provedených změn (kvůli nimž byl pull request uzavřen) nebo problém s review procesem (pull requesty čekají na provedení code review nebo na zapracování požadovaných změn). Při interpretaci této metriky je třeba brát ohled na nastavené procesy ve vývojovém týmu.
    \item \textbf{Velikost pull requestu} (aktivita)\\
    Počet přidaných a odebraných řádků a počet změněných souborů v rámci pull requestu. Pull requesty s velkým množstvím změn nebo se změnami ve velkém počtu souborů mohou být obtížné na kontrolu, a proto by například přidání velké funkcionality mělo být rozděleno na dílčí úkoly, které budou zapracovány samostatně.
    \item \textbf{Počet napsaných komentářů} (aktivita, komunikace a spolupráce)\\
    Počet napsaných komentářů u pull requestu, kde je daný uživatel přiřazený jako reviewer. Malý počet komentářů může naznačovat menší důkladnost provedeného code review. Hodnotu této metriky je potřeba opět dávat do souvislostí s nastavenými pravidly pro code review proces. V některých vývojových týmech například reviewer kontroluje kód pouze na chyby, v jiných dává i tipy na optimalizace kódu nebo jeho jiná vylepšení.
    \item \textbf{Počet komentářů od kolegů} (aktivita, komunikace a spolupráce)\\
    Počet napsaných komentářů u pull requestu, který uživatel vytvořil. Velký počet komentářů může indikovat, že kód obsahoval větší počet chyb nebo jiné nedostatky.
    \item \textbf{Rychlost mergnutí pull requestu} (výkon, komunikace a spolupráce)\\
    Čas od vytvoření pull requestu do doby jeho mergnutí. Pull requesty, které byly uzavřeny nebo které jsou otevřené, nejsou do této statistiky započítané.
    \item \textbf{Rychlost zahájení review procesu} (výkon)\\
    Čas od vytvoření pull requestu do doby přidání prvního komentáře od reviewera nebo do doby schválení pull requestu reviewerem, pokud žádný komentář nepřidal. Otevřené a uzavřené pull requesty, u kterých nedošlo ani k jedné z výše uvedených akcí, nejsou do této statistiky započítané.
\end{enumerate}

Některé z vybraných metrik úzce souvisí také s DORA metrikami uvedenými v sekci \ref{section:dora}. Rychlost zahájení review procesu a rychlost mergnutí pull requestu například ovlivňuje frekvenci nasazování či čas od commitnutí změn do jejich nasazení na produkčním prostředí. Velké pull requesty zvyšují riziko chyb, což se může negativně promítnout do míry selhání změn. Tu dále může ovlivnit také počet napsaných komentářů, který může naznačovat důkladnost provedeného code review.
