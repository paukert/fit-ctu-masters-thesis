\section{Metriky}
Následující podkapitola je věnována popisu metrik, pomocí nichž je možné porovnávat úroveň aktivity a výkonnost vývojářů v různých oblastech vývoje softwaru. S ohledem na cíle práce budou primárně přiblíženy metriky, pro které je možné získat data z nástrojů GitHub a GitLab.

\subsection{Aktivita u tasků}
Pojmem task je v tomto případě rozuměn sourhn informací, které obsahují popis požadovaných úprav, informaci o typu, stavu, prioritě a případně i dalších souvisejících věcech. Mezi základní typy tasků se řadí oprava chyby, přidání nové funkcionality, zlepšení existující funcionality a technický dluh. Množina stavů by měla minimálně obsahovat stav „otevřeno“ a „uzavřeno“, kde stav „otevřeno“ reprezentuje situaci, kdy všechny požadované úpravy ještě nebyly zapracovány nebo se čeká na kontrolu od ostatních vývojářů nebo testerů. Pokud všichni členové týmu dokončili práci na daném tasku, tak by se jeho stav měl změnit na „uzavřeno“. V nástrojích GitHub a GitLab se tasky většinou evidují ve formě issues, u kterých je možné diskutovat s ostatními uživateli pomocí komentářů.

Ve spojení s tasky je tedy mezi metriky možné zařadit počet založených a uzavřených tasků nebo počet napsaných komentářů. Některé z těchto metrik avšak nejsou pro porovnávání výkonnosti vývojářů ve většině případů moc vypovídající. Prvním z problémů by mohl být fakt, že vývojáři většinou nejsou autory tasků, neboť ty spíše zakládají analytici nebo produktoví manažeři. Podobný problém nastává u metriky počtu uzavřených tasků, protože ty většinou po otestování uzavírají testeři systémů. V neposlední řadě by také mohl být problém se získáním těchto dat v souvislosti se systémy GitHub a GitLab, poněvadž větší týmy často tasky evidují v jiných systémech, které obsahují pokročilejší funkcionality. \subsection{Úprava zdrojového kódu}
Jednou z prvních široce využívaných metrik pro porovnání velikosti systémů a produktivity vývojářů byl počet (upravených) řádků zdrojového kódu (LOC). \cite{LOC-parareda} Výhody této metriky zahrnovaly využitelnost u různých programovacích jazyků, nízkou časovou náročnost na její získání a například také jednoduchost vysvětlení této metriky i lidem z jiných oborů. Na druhou stranu má tato metrika ale i několik nevýhod, kvůli nimž je využívání počtu řádků jako primárního indikátoru produktivity vývojářů značně problematické. Počet řádků totiž nic neříká o tom, jak kvalitní je výsledný kód a také je možné, že stejné funkcionality by bylo možné dosáhnout s výrazně menším počtem řádků při využití jiných konstruktů programovacího jazyka. Pro zvýšení vypovídající hodnoty této metriky by bylo potřeba data předzpracovávat a například odebrat nedosažitelný kód, sjednotit formátování, ignorovat prázdné řádky a podobně. V neposlední řadě může být problematické porovnávání počtu řádků mezi projekty, které využívají odlišné programovací jazyky, poněvadž každý z jazyků bude pravděpodobně vývojářům poskytovat odlišné funkce a bude vyžadovat odlišný způsob zápisu zdrojového kódu.

\subsection{Code review}
Code review je proces, při kterém členové vývojového týmu kontrolují kód napsaný jinými vývojáři. Cílem tohoto procesu je zajistit dobrou kvalitu kódu -- minimalizovat počet případných chyb, zlepšit jeho čitelnost a konzistenci. Pro code review se často využívají nástroje jako GitHub a GitLab, které tento proces podporují svou funkcionalitou v rámci pull requestů\footnote{V GitLabu se místo pojmu „Pull request“ využívá pojem „Merge request“}.

V souvislosti s code review lze evidovat mimo jiné následující metriky:
\begin{itemize}
    \item počet vytvořených diskuzí (vláken)
    \item počet přidaných komentářů
    \item počet vyřešených diskuzí
    \item čas do mergnutí pull requestu, u kterého je uživatel jako reviewer
    \item čas do mergnutí uživatelova pull requestu
\end{itemize}
