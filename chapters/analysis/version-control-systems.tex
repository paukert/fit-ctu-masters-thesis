\section{Systémy pro správu verzí softwaru}
Systémy pro správu verzí softwaru (VCS) představují klíčový nástroj při vývoji jakéhokoliv softwaru. Kromě uchovávání historie změn zdrojového kódu umožňují mimo jiné efektivní spolupráci více vývojářů na jednom projektu a minimalizují riziko ztráty práce při selhání počítače.

Následující podkapitola se věnuje nástrojům pro správu verzí softwaru pomocí verzovacího systému Git. Tyto nástroje však v dnešní době nepodporují pouze samotnou správu verzí softwaru, ale nabízí i ticketovací systémy, možnost tvorby stránek s dokumentací, nástroje pro CI/CD a mnoho dalších souvisejících funkcionalit. V dalších sekcích budou popsány pouze nástroje GitHub a GitLab, ale mezi další zástupce patří například nástroj Bitbucket od firmy Atlassian. \todo{Rozmyslet upravení odstavce, přidat nějaké informace o gitu?}

\subsection{Základní pojmy}
V následující sekci budou představeny základní pojmy související se systémy pro správu verzí softwaru, které budou v práci využívány.

\begin{description}
    \item[Repozitář] Repozitář je úložiště, ve kterém se uchovávají všechny verze zdrojového kódu a souvisejících souborů projektu. Může být buď lokální (uložený na počítači vývojáře), nebo vzdálený (umístěný například na serveru GitHub nebo GitLab). \cite{pro-git}
    \item[Commit] Commit je operace, která zachycuje provedené změny v obsahu souborů do repozitáře. Je to základní jednotka práce ve verzovacích systémech. Díky commitům je možné sledovat kdy, co a proč se měnilo. \cite{pro-git} Důvod provedených změn by měl být zřejmý z titulku nebo popisu commitu viz sekce \ref{commit-message}, avšak někdy tomu tak nemusí být.
    \item[Větev] Větev představuje izolovaný pracovní prostor, ve~kterém lze vyvíjet nové funkcionality, opravovat chyby nebo testovat nové nápady bez ovlivnění hlavní linie vývoje. Využívání větví umožňuje efektivní a bezpečnou týmovou spolupráci, neboť každý člen vývojového týmu může pracovat ve své vlastní větvi. \cite{pro-git}
    \item[Pull request] Pull request je návrh na začlenění změn z jednoho nebo více commitů z jedné větve do větve jiné. Před zahrnutím změn do cílové větve je možné provádět code review. V GitLabu se místo pojmu „Pull request“ využívá pojem „Merge request“. \cite{pro-git} V rámci této práce bude dále jednotně využíván pojem „Pull request“.
    \item[Code review] Code review je proces, při kterém členové vývojového týmu kontrolují a píší připomínky nebo návrhy na zlepšení ke kódu napsaného jinými vývojáři. Cílem tohoto procesu je zajistit dobrou kvalitu kódu. Člověk, který provádí kontrolu se nazývá reviewer. \cite{code-review} Pro code review se často využívají nástroje jako GitHub a GitLab, které tento proces podporují svou funkcionalitou v rámci pull requestů.
    \item[Kvalitní kód] Kvalitní zdrojový kód je čitelný a snadno pochopitelný i pro~někoho, kdo ho nepsal. Používá smysluplné a konzistentní názvy, má jednotný styl formátování a je logicky strukturovaný do menších částí. Každá funkce nebo třída plní vymezený účel a kód se vyhýbá zbytečné složitosti. Je bez chyb, snadno udržovatelný, dobře testovatelný a doplněný přehlednou dokumentací, která usnadňuje jeho další rozvoj a pochopení. \cite{code-quality}
    \item[Pipeline] Jako pipeline se v kontextu CI/CD označuje sled kroků, které se automaticky provádí, když dojde k určité události. Typicky takovou událostí může být změna zdrojových kódů v repozitáři nebo například otevření pull requestu.
    \item[Úkol] Pojmem úkol je v kontextu této práce rozuměn souhrn informací, které obsahují popis požadovaných úprav, informaci o typu, stavu, prioritě a kategorizaci. Mezi základní typy úkolů se řadí oprava chyby, přidání nové funkcionality, zlepšení existující funcionality a technický dluh. Množina stavů by měla obsahovat minimálně stav „otevřeno“ a „uzavřeno“, kde stav „otevřeno“ reprezentuje situaci, kdy všechny požadované úpravy ještě nebyly zapracovány nebo se čeká na kontrolu od ostatních vývojářů nebo testerů. Pokud všichni členové týmu dokončili práci na daném úkolu, tak by se jeho stav měl změnit na „uzavřeno“. V nástrojích GitHub a GitLab se úkoly většinou evidují ve formě issues, u kterých je možné diskutovat s ostatními uživateli pomocí komentářů.
\end{description}

\subsection{GitHub}
Vývoj platformy GitHub začal v roce 2007 a od té doby její uživatelská základna neustále roste. Na začátku roku 2023 už GitHub využívalo více než 100~milionů uživatelů a 4 miliony organizací na správu softwarových projektů. \cite{GitHub, GitHub-about} Základní funcionality jsou dostupné zdarma jak u veřejných, tak u privátních repozitářů. Pokročilejší funkce, jako jsou automatické detekce úniků citlivých informací nebo nastavování pravidel pro zahrnování změn do chráněných větví, jsou ale dostupné pouze pro veřejné repozitáře nebo jsou součástí placených verzí. Self-hosted řešení je dostupné pouze v placené enterprise variantě. \cite{GitHub-pricing}

\subsection{GitLab}
GitLab začal být vyvíjen jako open source projekt v roce 2011 a od té doby se na jeho vylepšování podílelo více než 3 300 přispěvatelů. Aktuálně má GitLab dle odhadů přes 30 milionů registrovaných uživatelů, kteří si mohou stejně jako u GitHubu vybrat z několika nabízených cenových plánů. \cite{GitLab-about} Díky komunitní open source verzi ale GitLab nabízí i bezplatnou self-hosted variantu, která organizacím umožňuje mít všechna data pod svou kontrolou. \cite{GitLab-pricing}

\subsection{API}
GitHub i GitLab poskytují REST a GraphQL API, pomocí nichž lze provádět nejrůznější integrace, vytvářet nástroje na automatizaci procesů nebo mohou sloužit jako zdroj dat pro analytické dashboardy. Dokumentace těchto API jsou dostupné online na webových stránkách GitHubu \cite{GitHub-REST, GitHub-GraphQL} a GitLabu \cite{GitLab-REST, GitLab-GraphQL}.

\begin{description}
    \item[REST] Representational State Transfer je architektonický styl pro distribuované hypermediální systémy. Je založen na architektuře klient--server. Samotný server si však mezi jednotlivými požadavky neuchovává žádný kontext tj. požadavek od klienta musí vždy obsahovat veškeré informace, které jsou potřebné pro jeho zpracování. Kromě bezstavovosti se mezi další principy REST řadí uniformní rozhraní založené na zdrojích a možnost ukládání dat do mezipaměti. \cite{REST} Ačkoliv REST není vázán na konkrétní komunikační protokol, tak se v dnešní době využívá především s protokolem HTTP. \cite{REST-HTTP}
    \item[GraphQL] GraphQL je dotazovací jazyk, který originálně vznikl v roce 2012 pro potřeby společnosti Facebook (dnes Meta Platforms). Návrh specifikace GraphQL byl v roce 2015 vydán pod open source licencí a od roku 2019 se o ni stará GraphQL Foundation. Pomocí GraphQL je možné stejně jako pomocí REST API číst, vytvářet a libovolně modifikovat uložené záznamy. \cite{GraphQL} Jako komunikační protokol je opět nejčastěji využíván HTTP. \cite{GraphQL-HTTP} GraphQL je založen na typech a jedna z jeho hlavních výhod spočívá v tom, že si klient sám určuje, která data se mu mají v odpovědi zaslat. \cite{GraphQL}
\end{description}

Na výpisech kódu \ref{listing:rest-query-example} a \ref{listing:graphql-query-example} je znázorněno, jak lze pomocí REST a GraphQL API u nástroje GitHub získat popis repozitáře a název prvních deseti issues pro~projekt uživatele \mintinline{text}|novak| s názvem \mintinline{text}|kalkulacka|. V případě REST API by bylo třeba provést dva samostatné GET požadavky -- první na získání popisu repozitáře a druhý pro získání názvů issues viz výpis kódu \ref{listing:rest-query-example}. Při využití GraphQL API naopak stačí provést pouze jeden POST požadavek na URL \mintinline{text}|https://api.github.com/graphql| s tělem, které je vidět na výpisu kódu \ref{listing:graphql-query-example}. I na tomto malém příkladu je vidět, že využití GraphQL API může být v mnoha případech efektivnější v porovnání s REST API. Efektivita nespočívá pouze v menším počtu požadavků, ale také v úspoře objemu přenášených dat, protože GraphQL API na rozdíl od REST API poskytuje klientovi pouze ty informace, o které si výslovně požádá.
 
\begin{listing}[h]
    \caption{Získání informací o repozitáři a issues pomocí REST API}\label{listing:rest-query-example}
    \begin{minted}[breaklines, breakbefore=?]{bash}
curl https://api.github.com/repos/novak/kalkulacka
curl https://api.github.com/repos/novak/kalkulacka/issues?per_page=10
    \end{minted}
\end{listing}

    \caption{Získání informací o repozitáři a issues pomocí GraphQL API}\label{listing:graphql-query-example}

  repository(owner: "novak", name: "kalkulacka") {
    description
    issues(first: 10) {
      nodes {
        title
      }
    }
  }
}
    \end{minted}

\subsubsection{Autentizace a autorizace}
U veřejných repozitářů služeb GitHub i GitLab je umožněno provádět vybrané akce i bez autentizace. Vzhledem k nižším limitům počtu požadavků pro neautentizované uživatele a zajištění funkčnosti i u soukromých repositářů je však obvykle nezbytné zasílat autentizační údaje při každém požadavku.

Pro autentizaci požadavku na GitLab REST API (verze 4) a GraphQL API je možné využít některý z následujících způsobů:
\begin{itemize}[topsep=4pt, itemsep=2pt]
  \item session cookie -- využívá se převážně u samotné webové GitLab aplikace;
  \item osobní přístupový token -- je vázaný na konkrétní uživatelský účet;
  \item projektový přístupový token -- je vázaný na konkrétní projekt. Projektové přístupové tokeny jsou podporovány pouze v placených plánech nebo~na~self-hosted instancích;
  \item skupinový přístupový token -- je vázaný na skupinu projektů. Skupinové tokeny jsou opět podporovány pouze v placených plánech nebo~na~self-hosted instancích;
  \item OAuth 2.0 token.
\end{itemize}

Přístupový token je možné zasílat buď v hlavičkách požadavku, nebo pomocí query parametru. Všem typům tokenů lze přiřadit specifický rozsah oprávnění, který určuje, jaké akce mohou být s využitím daného tokenu prováděny. Tímto způsobem lze omezit přístup pouze na nezbytné operace a snížit riziko případného zneužití. \cite{GitLab-REST-auth, GitLab-auth}

U služby GitHub lze autentizaci jak pro REST API (verze 2022-11-28), tak pro GraphQL API provést pomocí následujících způsobů:
\begin{itemize}[topsep=4pt, itemsep=2pt]
  \item osobní přístupový token -- je určený pro osobní použití. GitHub umožňuje nastavení velice konkrétních oprávnění, která mohou být navíc omezena pouze na konkrétní projekty;
  \item GitHub App;
  \item OAuth App.
\end{itemize}

Autentizace pomocí GitHub App a OAuth App se používá v případech, kdy aplikace potřebuje přistupovat k API samostatně (např. jménem organizace nebo konkrétního uživatele). Obě možnosti poskytují podobný způsob autentizace, ale je doporučeno spíše využívat GitHub Apps, které mimo jiné umožňují detailnější nastavení konkrétních oprávnění a využívají tokeny s krátkou délkou platnosti. \cite{GitHub-REST-auth, GitHub-auth, GitHub-oauth-apps}

GitHub Apps mohou využívat dva hlavní způsoby autentizace při přístupu k REST a GraphQL API. První možností je autentizace jako nainstalovaná aplikace a druhou autentizace jménem koncového uživatele. Autentizace jako nainstalovaná aplikace umožňuje přístup ke zdrojům, které jsou vlastněny uživatelem nebo organizací, kteří si GitHub App nainstalovali. Tato možnost je ideální pro nejrůznější automatizace, kdy není vyžadován vstup uživatele. Autentizace na API v tomto případě probíhá pomocí tzv. krátkodobého instalačního přístupového tokenu, který GitHub App získá pomocí vygenerovaného JWT. Při využití autentizace jménem konkrétního uživatele využívá GitHub App OAuth uživatelský přístupový token, který aplikaci umožňuje provádět akce v rozsahu oprávnění, jež uživatel schválil. Pokud GitHub App využívá autentizaci jménem uživatele, tak jsou prováděné aktivity přiřazeny k danému uživateli. V opačném případě bude jako autor aktivit označena konkrétní GitHub App. \cite{GitHub-GitHub-app}
