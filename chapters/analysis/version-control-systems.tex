\section{Nástroje pro správu verzí softwaru}
Následující podkapitola se věnuje nástrojům pro správu verzí softwaru pomocí verzovacího systému Git. Tyto nástroje však v dnešní době nepodporují pouze samotnou správu verzí softwaru, ale nabízí i ticketovací systémy, možnost tvorby stránek s dokumentací, nástroje pro CI/CD a mnoho dalších souvisejících funkcionalit. V dalších sekcích budou popsány pouze nástroje GitHub a GitLab, ale mezi další zástupce patří například nástroj Bitbucket od firmy Atlassian.

\subsection{GitHub}
Vývoj platformy GitHub začal v roce 2007 a od té doby její uživatelská základna neustále roste. Na začátku roku 2023 už GitHub využívalo více než 100 milionů uživatelů a 4 miliony organizací na správu softwarových projektů. \cite{GitHub, GitHub-about} Základní funcionality jsou dostupné zdarma jak u veřejných, tak u privátních repozitářů. Pokročilejší funkce jako jsou automatické detekce úniků tokenů a hesel nebo nastavování pravidel pro zahrnování změn do chráněných větví, jsou ale dostupné pouze pro veřejné repozitáře nebo jsou součástí placených verzí. Self-hosted řešení je dostupné pouze v placené enterprise variantě. \cite{GitHub-pricing}

\subsection{GitLab}
GitLab začal být vyvíjen jako open source projekt v roce 2011 a od té doby se na jeho vylepšování podílelo více než 3300 přispěvatelů. Aktuálně má GitLab dle odhadů přes 30 milionů registrovaných uživatelů, kteří si mohou stejně jako u GitHubu vybrat z několika nabízených cenových plánů. \cite{GitLab-about} Díky komunitní open source verzi ale GitLab nabízí i bezplatnou self-hosted variantu, která organizacím umožňuje mít všechna data pod svou kontrolou. \cite{GitLab-pricing}

\subsection{APIs}
GitHub i GitLab poskytují REST a GraphQL APIs, pomocí nichž lze provádět nejrůznější integrace, vytvářet nástroje na automatizaci procesů nebo mohou sloužit jako zdroj dat pro analytické dashboardy. Dokumentace těchto APIs jsou dostupné online na webových stránkách GitHubu \cite{GitHub-REST, GitHub-GraphQL} a GitLabu \cite{GitLab-REST, GitLab-GraphQL}. Ačkoliv je u veřejných repozitářů možné provádět některé akce bez autentizace, tak je s ohledem na nižší limity pro neautentizované uživatele většinou nutné zasílat informaci o autentizaci při každém požadavku. Dostupné způsoby autentizace pro služby GitHub a GitLab budou popsány dále.

\begin{description}
    \item[REST] Representational State Transfer je softwarová architektura pro distribuované hypermediální systémy. Je založena na vrstvené architektuře klient -- server. Mezi hlavní pricipy RESTu patří bezstavovost (server si neuchovává žádný kontext mezi jednotlivými požadavky tj. požadavek od klienta musí vždy obsahovat veškeré informace, které jsou potřebné pro jeho zpracování), uniformní rozhraní založené na zdrojích a cachovatelnost. \cite{REST} Ačkoliv REST není vázán na konkrétní komunikační protokol, tak se v dnešní době využívá především s protokolem HTTP. \cite{REST-HTTP}
    \item[GraphQL] GraphQL je dotazovací jazyk, který originálně vznikl v roce 2012 pro potřeby společnosti Facebook (dnes Meta Platforms). Návrh specifikace GraphQL byl v roce 2015 vydán pod open source licencí a od roku 2019 se o ni stará GraphQL Foundation. Pomocí GraphQL je možné stejně jako pomocí REST API číst, vytvářet a libovolně modifikovat uložené záznamy. \cite{GraphQL} Jako komunikační protokol je opět nejčastěji využíván HTTP. \cite{GraphQL-HTTP} GraphQL je založen na typech \cite{GraphQL} a jedna z jeho hlavních výhod spočívá v tom, že si klient sám určuje, která data se mu mají v odpovědi zaslat.
\end{description}

Na následujícím příkladu je vidět, jakým způsobem lze pomocí REST a GraphQL API u nástroje GitHub získat popis repozitáře \mintinline{text}|kuoris-v2| a název prvních deseti issues. V případě REST API by bylo třeba provést dva GET požadavky -- první na získání popisu repozitáře a druhý pro získání názvů issues viz výpis kódu \ref{listing:rest-query-example}. Při využití GraphQL API stačí provést jeden POST požadavek na \mintinline{text}|https://api.github.com/graphql| s tělem, které je vidět i s obrženou odpovědí na výpisu kódu \ref{listing:graphql-query-example}. I na tomto malém příkladu je vidět, že využití GraphQL API může být v mnoha případech efektivnější v porovnání s REST API. Efektivnost nespočívá pouze v menším počtu požadavků, ale i ve velikosti přenášených dat, neboť jak již bylo dříve zmíněno, tak v případě GraphQL jsou klientovi zasílány pouze informace, o které si sám zažádá.
 
\begin{listing}[h]
    \caption{Ukázka získání informací o repozitáři a issues pomocí REST API}\label{listing:rest-query-example}
    \begin{minted}[breaklines, breakbefore=?]{bash}
curl https://api.github.com/repos/paukert/kuoris-v2
curl https://api.github.com/repos/paukert/kuoris-v2/issues?per_page=10
    \end{minted}
\end{listing}

\begin{listing}[h!]
    \caption{Ukázka těla požadavku a odpovědi z GraphQL API}\label{listing:graphql-query-example}
    \begin{minted}{graphql}
query {
  repository(owner: "paukert", name: "kuoris-v2") {
    description
    issues(first: 10) {
      nodes {
        title
      }
    }
  }
}
    \end{minted}
    \begin{minted}{json}
{
  "data": {
    "repository": {
      "description": "Bachelor's Thesis, FIT CTU",
      "issues": {
        "nodes": [
          {
            "title": "Docker-compose pro celou aplikaci"
          },
          {
            "title": "Vytvořit šablony pro chybové stránky"
          }
        ]
      }
    }
  }
}
    \end{minted}
\end{listing}

Pro autentizaci požadavku na GitLab REST a GraphQL API lze využít session cookie nebo se lze autentizovat pomocí osobního, projektového\footnote{Projektové přístupové tokeny jsou podporovány pouze v placených plánech nebo na self-hosted instancích.}, skupinového\footnote{Skupinové přístupové tokeny jsou dostupné pouze v placených plánech.} případně OAuth 2.0 tokenu, který musí mít nastaven správný scope. Přístupový token je možné zaslat v hlavičkách požadavku nebo pomocí query parametru. \cite{GitLab-REST-auth, GitLab-auth} U služby GitHub lze autentizaci jak pro REST API, tak pro GraphQL API provést pomocí osobního přístupového tokenu, případně pomocí tokenu vygenerovaného s využitím GitHub App. \cite{GitHub-REST-auth, GitHub-auth}
